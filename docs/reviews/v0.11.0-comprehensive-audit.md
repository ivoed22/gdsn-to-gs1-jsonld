possible GDSN/BMS/XPath source fields for GS1 Web Vocabulary properties.  The

accepted or written, no mapping YAML is modified, no external APIs are called,
and no converter logic is changed.

All hard rules are complied with.  The release is release-ready with minor
documentation and wording gaps fixed during this audit.

## Candidate Scoring Findings (Quality Assessment)

### Scoring is deterministic

Verified: same inputs always produce same outputs (test_output_is_deterministic
passes).

### Score weights are reasonable

- `existing_mapping_catalog_match` (0.90): correctly highest — already
  cataloged mappings should surface at top.
- `semantic_resource_urn_match` (0.80): appropriate for URN direct matches.
- `mapping_yaml_canonical_field_match` (0.65): slightly lower than URN match
  (acceptable — YAML fields are implementation details).
- `exact_property_name_match` (0.70): appropriate.
- Token overlap signals (0.15–0.35): suitably modest.
- `quantity_uom_compatible` (0.20): useful supplementary signal.
- `standards_review_linked` (0.10): small nudge to surface SDR-linked pairs.

### Score cap

Scores are capped in [0.0, 1.0].  Multiple signals can sum over 1.0 (e.g.
catalog match + URN match = 1.70) but the cap prevents inflation artifacts.

### Deleted rows

Deleted rows receive `deleted_attribute_warning` penalty (-0.30) and are
assigned `not_recommended` review status.  They do not silently appear as
normal high-confidence candidates.

### Class/Module rows

Non-attribute rows receive `class_row_not_attribute` penalty (-0.20).

### Quantity signal not overpowered

`quantity_uom_compatible` (0.20) is lower than all name-match signals.  It
cannot push a weak candidate to high confidence on its own.

### Limit per property works

Tested: limit=5 returns exactly 5 candidates (test_limit_per_property_respected
passes).

### Confidence classification is stable

`classify_confidence` uses fixed thresholds (0.70/0.40/0.15) applied
deterministically.  The `standards_review_linked` override forces
`review_required` for low-score SDR-linked pairs, which is correct.

## Remaining Recommendations for v0.11.1 or v0.12.0

1. **Scoring tuning** — After first human review pass of generated candidates,
   tune signal weights based on which signals produced useful vs. noisy results.

2. **Confidence threshold review** — The 0.70/0.40/0.15 thresholds are starting
   points.  A review of candidate quality across several properties will inform
   whether these need adjustment.

3. **GTIN duplicate candidates** — gs1:gtin returns 20 candidates all at score
   1.0 and all `already_mapped`.  This is because multiple GDSN rows share the
   `gtin` attribute name.  A deduplication or grouping step in a future version
   could reduce noise for already-mapped properties.

4. **All-properties run time** — The full all-properties run over 553 properties
   x 7171 GDSN rows is I/O-bound.  A future optimization (pre-built token index,
   early-exit for high-score pairs) could improve CLI performance for large runs.

5. **XLSX report** — openpyxl is not a mandatory dependency; the XLSX download
   silently degrades to an info message.  Consider making the requirement
   explicit in docs or adding openpyxl to required dependencies in a future
   release.

6. **Streamlit empty state for no inputs loaded** — If reference CSVs are
   missing, the workflow shows a generic `st.error()`.  A more actionable
   message pointing to the `import-reference-data` CLI command would help users.
