# v0.12.0 — Product Passport Source Import & Schema Validator

## Overview

Version v0.12.0 adds the Product Passport Bridge, a prototype/reference workflow
that inventories public Digital Product Passport (DPP) reference sources and
validates prototype Product Passport JSON against local JSON Schemas.

This release does not claim official GS1 validation or production compliance.
All Product Passport work is clearly labelled as prototype/reference.

Converter logic, batch behavior, mapping YAML, mapping catalog data, and Web
Vocabulary snapshots are unchanged.

## What's New

### Product Passport Bridge

A new prototype/reference feature family: **Product Passport Bridge**.

**Feature scope for v0.12.0:**
- Source manifest tracking for DPP reference sources
- Source inventory CLI command and Streamlit tab
- Structural JSON Schema validation CLI command and Streamlit tab
- Examples inventory tab

### Source Manifest

File: `product_passport/reference_sources/source_manifest.json`

Tracks 7 source entries:
- 1 core DPP JSON-LD context (placeholder)
- 3 DPP JSON Schemas for general product, battery, and textile (placeholders)
- 1 SHACL shapes file for core DPP (placeholder, inventoried only — not executed)
- 1 minimal DPP example (committed prototype)
- 1 EPCIS DPP example (placeholder)

Placeholder entries require the user to download public source files and update
`sha256`, `retrieved_at`, and `source_url` with actual values.

The manifest JSON Schema is at:
`product_passport/reference_sources/source_manifest.schema.json`

### Built-in Minimal Example and Schema

Two small files are committed to the repository for testing:

- `product_passport/examples/minimal_product_passport.json` — A minimal
  prototype Product Passport JSON-LD object for structural testing. Not official
  GS1 or DPP production data.
- `product_passport/reference_sources/raw_public/schemas/dpp_minimal.schema.json` —
  A minimal JSON Schema that requires `@context` and `@type` to be present.
  Prototype/reference only, not an official DPP production schema.

### Structural Validation

The Schema Validator uses `jsonschema` (Draft7Validator) for structural
validation. If `jsonschema` is not installed, a minimal required-field fallback
is used with a warning.

This is structural validation only. It does not claim official GS1 validation
or EU DPP regulatory compliance.

### CLI Commands

New command: `gdsn-to-gs1-jsonld inventory-product-passport-sources`

```bash
gdsn-to-gs1-jsonld inventory-product-passport-sources \
  --manifest product_passport/reference_sources/source_manifest.json \
  --output-dir product_passport/reference_sources/normalized/
```

New command: `gdsn-to-gs1-jsonld validate-product-passport`

```bash
gdsn-to-gs1-jsonld validate-product-passport \
  --input product_passport/examples/minimal_product_passport.json \
  --schema product_passport/reference_sources/raw_public/schemas/dpp_minimal.schema.json \
  --output-dir product_passport/validation_reports/
```

### Streamlit Workflow

A sixth workflow card is added to the Streamlit app:

- Marker: PP
- Title: Validate Product Passport Sources
- Description: Inventory public Product Passport reference sources and validate
  prototype Product Passport JSON against local schemas.
- Outcome: Reference-source inventory and structural validation report. No
  production compliance claim.

The workflow is displayed in a 3+3 grid layout.

The workflow has three tabs:
1. Source Inventory — load manifest, view counts, browse table, download reports.
2. Schema Validator — upload/paste JSON, select schema, validate, download report.
3. Examples — list and preview example entries.

## Note on jsonschema Dependency

`jsonschema` is installed as part of the v0.12.0 environment (version 4.x).
It is not yet listed in `pyproject.toml` as a required dependency because it
may not be needed by all users. The validator gracefully falls back to a
minimal required-field check if `jsonschema` is not available.
