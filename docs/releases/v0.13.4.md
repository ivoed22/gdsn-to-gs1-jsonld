# v0.13.4 — Workspace Polish & Manual Builder Coverage

## Overview

Version v0.13.4 bundles the post-v0.13.3 workspace UI fixes and a large
expansion of the Manual JSON-LD Prototype Builder's property coverage.

Everything here is UI/config plus the manual Builder serializer. The converter,
batch behavior, single-file output, mapping YAML, mapping catalog, and Web
Vocabulary snapshots are unchanged. Manual Builder output stays prototype and is
not BMS/XPath traceable. No official GS1 validation or production compliance is
claimed.

## Workspace UI fixes

- **Wider workspace.** The main column is widened (~92rem) via both the
  `stMainBlockContainer` hook and the stable `.block-container` class, so content
  fills the space next to the sidebar instead of floating as a narrow centred
  block.
- **Equal-size cards.** Route cards and workflow cards use a firm min-height plus
  a flex column that pins the Outcome line to the bottom, so cards in a row are
  the same height and their buttons align. Card-row buttons share a min-height.
- **Readable active button.** Streamlit dims disabled buttons, which made the
  dark-blue "Active" label hard to read; it is now full-opacity with white text.

## Manual JSON-LD Builder — coverage expansion

The Builder manifest grew from ~40 fields across 10 groups to **183 fields
across 19 groups** — the large majority of the safely-authorable
`gs1:Product` / `gs1:FoodBeverageTobaccoProduct` properties. All codes and labels
are sourced from the local `webvoc/current` snapshot; nothing is invented.

### Breadth — more flat fields

Product descriptions & marketing, consumer information (instructions, safety,
recall), lifecycle dates, additional measurements (weights, areas, volumes,
dimensions), identifiers & variants (batch/lot, serial, production variant),
food serving details, and consumer/DPP link types.

### Depth — nested objects

A generic `object` input type renders an object's safe sub-fields and emits a
typed nested object:

- `gs1:brand` → `gs1:Brand`
- `gs1:image` / `gs1:referencedFile` / `gs1:instructionsForUse` /
  `gs1:consumerHandlingStorage` / `gs1:audioFile` → `gs1:ReferencedFileDetails`
- `gs1:certification` → `gs1:CertificationDetails`
- `gs1:packagingMaterial` → `gs1:PackagingMaterial`
- `gs1:hasAllergen` → `gs1:AllergenDetails`

### Controlled codes

A generic `code` input type offers a dropdown and emits the selection as a
JSON-LD node reference (`{"@id": "gs1:…"}`): allergen types & containment,
nutritional claim, preservation technique, growing method, source animal,
packaging accreditation / free-from / diet-allergen, and more.

### Nutrition

All 43 per-nutrient measurements (energy, fat, saturated fat, carbohydrates,
sugars, protein, salt, fibre, vitamins, minerals, …) plus the nutrient basis
quantity, emitted as `value` + `unitCode` quantities like `gs1:netContent`.

## Intentionally not mapped (reported, not changed)

- **Object references** to `gs1:Organization` (brandOwner, manufacturer) and
  `gs1:Country` (countryOfOrigin) — need a reference pattern; Country codes are
  not present as individuals in this snapshot.
- **Deeper `*Details` object classes** (ingredient, packaging, additive,
  organicClaim, targetMarket, …) — object-in-object structures.
- **Product-to-product references** and organisation/place-oriented link types.
- **Governed converter mapping YAML and mapping catalog** — changing these is a
  reviewed standards decision, not a builder/UI change.

## Preserved

- Converter logic, batch behavior, and single-file output are unchanged.
- Mapping YAML, mapping catalog, and Web Vocabulary snapshots are unchanged.
- No warnings suppressed; no mock data; no fabricated coverage/compliance.
- No GS1 ↔ Product Passport Crosswalk, SHACL execution, VC, or signed
  credentials.
